# M3.5 Code-action + 可插拔安全/沙箱 代码评审报告

> 日期：2026-06-12
> 评审对象：M3.5 native code-action + permission/sandbox 实施后的代码
> 依据文档：`plan/M3.5-CodeAction-Sandbox-plan.md`、`docs/Python复刻Pi-Roadmap-v1.md`
> 评审方式：源码阅读 + 全量测试 + 针对权限绕过 / code-action timeout / sandbox 语义的离线复现

## 1. 评审结论

**结论：M3.5 主体能力已落地，但安全/沙箱语义目前不能作为约束边界使用，建议先修 2 个 P1。**

当前实现已经完成 M3.5 的主要功能：

- `--code` 默认关闭，开启后注册 native `code` 工具。
- `code` 工具可在一次调用中用 Python 控制流组合多次 `mu.read/write/edit/bash/call`。
- `--permission allow|readonly|workspace` 已接到 `ToolRegistry.execute` 单一入口。
- `--sandbox local|docker` 已抽出 `Environment` Protocol，并提供实验性 `DockerEnvironment`。
- CLI / TUI / prompt / README / roadmap 都已接线。
- M2/M3 上轮评审中的两个 P1 已修：扩展崩溃会快速降级并注销工具；同名扩展加载会拒绝。

全量测试通过：

```text
65 passed, 1 skipped in 6.59s
```

但 M3.5 是“让 agent 更能动”的阶段，安全语义需要更严格。目前 `readonly`、`workspace`、`docker` 都容易被误解为真实约束边界，而实现实际上只覆盖了部分工具名或部分执行面。

## 2. Findings

### P1. 权限层对 M3.5 新执行面是可绕过的

- 位置：`mu/permission.py:21`-`26`
- 位置：`mu/permission.py:29`-`44`
- 位置：`mu/tools.py:225`-`231`
- 相关能力：`--permission readonly|workspace`、code-action、extension、tool gate

当前 permission 策略只按工具名做最小判断：

```python
def read_only(name: str, args: dict[str, Any]) -> str | None:
    if name in _MUTATING:
        return f"{name} is blocked in read-only mode"
    if name == "bash":
        return "bash is blocked in read-only mode"
    return None
```

```python
def workspace_write(root: str) -> PermissionPolicy:
    ...
    if name in _MUTATING:
        ...
```

这能拦住内置 `write/edit/bash` 的一部分调用，但挡不住 M3/M3.5 新增执行面：

1. `readonly` 下，`code` 工具可直接用 Python `open(..., "w")` 写文件。
2. `readonly` 下，`load_extension` 可执行任意 Python 扩展文件；扩展顶层代码在 manifest 输出前就已经运行。
3. `readonly` 下，扩展工具即使声明 `permissions=["write"]`，仍会被放行，因为 core 没有把 manifest permission 接入 policy。
4. `workspace` 下，`bash` 可写 workspace 外路径，因为该策略只检查 `write/edit` 的 `path`。

离线复现结果：

```text
readonly_out= done
readonly_file_exists= True bypass

out= Loaded extension side v0.1; new tools:
target_exists= True loaded

out= wrote
target_exists= True bypass

workspace_bash_out= [exit code: 0]
workspace_bash_outside_exists= True bypass
```

**影响：**

- `--permission readonly` 容易被用户理解为不会写文件，但当前 code-action / extension 都可写。
- `--permission workspace` 容易被理解为写操作限制在 workspace 内，但 `bash` 可越界。
- M3.5 roadmap 中“权限层 gate 内置 / 扩展 / code-action 内层调用”的说法目前只覆盖“经 `mu.*` 或 registry 的工具调用”，没有覆盖 code-action 自身 Python I/O、扩展顶层代码和扩展内部副作用。

**建议：**

把 permission 从“工具名黑名单”升级为“capability gate”：

1. 给工具注册元数据增加 capability，例如 `read` / `write` / `shell` / `code_exec` / `extension_exec` / `network`。
2. `read_only` 默认拒绝 `write`、`shell`、`code_exec`、`extension_exec`，除非用户显式放行。
3. `workspace` 对 `write/edit` 做 path enforcement，同时对 `bash`、`code`、`extension` 明确拒绝或标注无法保证 workspace 约束。
4. 扩展 manifest 的 `permissions` 不再 advisory；注册扩展工具时把 permissions 存入 registry 元数据，并由 policy 使用。
5. `load_extension/reload_extension` 在 restrictive policy 下默认拒绝，因为加载扩展本身就是执行任意 Python。

如果阶段目标仍是“默认关、实验性”，也建议至少在 CLI/README 输出里把 `readonly/workspace` 改名或标注为“工具级软限制，不覆盖 code/extension 直接副作用”。

### P1. code-action 超时后 worker thread 仍会继续执行副作用

- 位置：`mu/codeact.py:89`-`98`
- 位置：`mu/codeact.py:99`-`113`
- 相关能力：native code-action timeout、可取消性、执行隔离

当前 code-action 通过 `asyncio.to_thread()` 在线程里执行模型写的 Python：

```python
return await asyncio.wait_for(
    asyncio.to_thread(self._run, code, loop), timeout=_CODE_TIMEOUT
)
```

`wait_for` 只能让外层 coroutine 返回超时，无法杀掉 Python worker thread。离线复现将 `_CODE_TIMEOUT` 暂设为 `0.2`，执行代码先 sleep 再写文件：

```text
timeout_out= Error: code execution timed out (worker thread may still be running).
exists_immediate= False
exists_later= True content= late
```

也就是说，工具结果已经返回“超时”，但旧代码仍在后台继续执行并产生副作用。

**影响：**

- agent 可能进入下一轮时，上一轮 code-action 仍在修改 workspace。
- 用户看到 timeout 可能以为动作已停止，但实际副作用仍在发生。
- 若代码里继续调用 `mu.*`，还可能与后续工具调用交错，污染事件流和 session 语义。

**建议：**

更稳妥的方向是把 code-action 改为可终止的子进程执行：

1. 子进程执行 code-action，超时后 kill process group。
2. 子进程与 core 用 JSONL/RPC 调 `mu.*`，仍走 `ToolRegistry.execute`。
3. 这样 timeout、权限、事件观测都能保持可控。

如果坚持进程内 exec，需要明确它不是可杀的隔离执行：

- timeout 文案改为 “soft timeout; code may still be running”。
- 超时后禁用该 `_MuApi` 实例继续发起 `mu.*`。
- 对 direct Python I/O 仍只能文档化风险，无法硬停。

### P2. Docker sandbox 语义不完整，容易被误认为真实沙箱

- 位置：`mu/environment.py:99`-`128`
- 位置：`mu/environment.py:130`-`134`
- 相关能力：`--sandbox docker`、Environment provider、文件工具与 bash 隔离边界

`DockerEnvironment` 当前只把 `bash` 放进 Docker：

```python
docker run --rm -v {workspace}:/workspace -w /workspace {image} sh -c {command}
```

但文件 IO 仍委托宿主：

```python
async def read_file(...):
    return await self._local.read_file(...)

async def write_file(...):
    await self._local.write_file(...)
```

离线复现：

```text
outside_write_exists= True outside
workspace= /.../ws
```

也就是说，即使 `DockerEnvironment(workspace=ws)`，`write_file(outside, ...)` 仍能写 workspace 外宿主路径。

另外，`DockerEnvironment` 注释写“隔离网络/进程”，但 `docker run` 未设置 `--network none`。Docker 默认 bridge network，不能称作网络隔离。

**影响：**

- `--sandbox docker` 目前只约束 bash 的进程环境，不约束 read/write/edit。
- README 中“bash 在容器内执行”是准确的，但 `environment.py` 注释“隔离网络/进程”过强。
- 用户可能误以为选择 docker 后所有工具都在沙箱内。

**建议：**

二选一收敛语义：

1. 保守文档化：明确 `DockerEnvironment` 仅 sandbox `bash`，文件工具仍是宿主 IO。
2. 强化实现：对 `read_file/write_file` 增加 workspace path enforcement，并把容器内 `/workspace/...` 与宿主 workspace 做双向映射。

如果继续宣称网络隔离，`docker run` 应至少加：

```text
--network none
```

并补测试覆盖：

- docker bash 不能访问网络。
- docker env 文件写不能越过 workspace。
- docker timeout 后容器被清理。

### P3. 包元数据仍停在 M3

- 位置：`pyproject.toml:8`

README 和 roadmap 已更新到 M3.5，但 package description 仍是：

```text
μ (mu) — a minimal, Pi-style async coding agent (M3 self-extension)
```

**影响：**

- 不影响运行，但 editable install / package metadata 会误导后续归档。

**建议：**

同步为 M3.5，例如：

```text

```

## 3. 正向观察

1. **默认关闭做得对**：`code_action=False`，`permission=allow`，`sandbox=local`，默认行为可退回 M3 形态。
2. **接线位置合理**：permission 挂在 `ToolRegistry.execute`，所有经 registry 的调用都会被统一 gate。
3. **code-action 的收益验证方向正确**：一次 `code` 调用中循环读 3 个文件，能体现相对多轮 tool call 的 round-count 收益。
4. **sync ↔ async 桥实现简洁**：`run_coroutine_threadsafe` 让模型代码可以用同步 `mu.read()` 组合 async 工具，API 体验好。
5. **M2/M3 遗留 P1 已修复**：扩展崩溃降级、同名扩展拒绝加载都已经有测试覆盖。
6. **TUI 扩展事件补齐**：`ExtensionLoaded/Unloaded/Log/Error` 已在 `TuiRenderer` 中渲染。

## 4. 验证结果

自动化测试：

```text
65 passed, 1 skipped in 6.59s
```

额外离线复现：

- `readonly` 下 code-action 直接 Python `open()` 可写文件。
- `readonly` 下 `load_extension` 可运行扩展顶层副作用。
- `readonly` 下扩展工具可写文件，manifest `permissions=["write"]` 未参与 enforcement。
- `workspace` 下 `bash` 可写 workspace 外路径。
- code-action timeout 返回后，worker thread 仍会继续写文件。
- `DockerEnvironment.write_file()` 可写 workspace 外宿主路径。

## 5. 建议处理顺序

1. **先修 P1-a**：明确 `readonly/workspace` 对 `code`、`load_extension`、扩展工具、`bash` 的策略语义，并升级为 capability-based gate。
2. **再修 P1-b**：code-action timeout 改成可终止子进程，或至少降级为明确的 soft timeout 风险。
3. **修 P2**：收敛 DockerEnvironment 的 sandbox 语义；要么仅宣称 bash sandbox，要么对文件 IO 做 workspace enforcement。
4. **顺手修 P3**：同步 `pyproject.toml` description。

## 6. 进入 v2 / M4 前的建议门禁

建议满足以下条件后再进入 v2：

- `--permission readonly` 下不会通过 code-action / extension / bash 产生写副作用，或 CLI/README 明确声明这些执行面被禁用。
- `--permission workspace` 下所有允许的写副作用都限定在 workspace 内；无法保证的工具默认拒绝。
- code-action 超时后不会继续产生副作用。
- Docker sandbox 的文档语义与实现一致。
- 新增覆盖上述绕过路径的回归测试。
