## P1 — 应修复

### P1-1 `redact_secrets` 名称匹配放宽到子串 + 全量值替换 → 可能损坏产物

`mu/eval.py:110-120`，本轮 diff 把：

```python
-        if "API_KEY" in upper or upper.endswith("_TOKEN") or upper.endswith("_SECRET"):
+        if "API_KEY" in upper or "TOKEN" in upper or "SECRET" in upper:
             out = out.replace(value, f"[REDACTED:{name}]")
```

**问题**：名称从「精确后缀」放宽为「子串包含」，而值替换是无条件 `str.replace(value, ...)`（全量、无最小长度护栏）。这会命中一批**名字含 TOKEN/SECRET 但值是普通短串**的常见环境变量，并把该短值在所有产物里全局替换掉：

- `TOKENIZERS_PARALLELISM=false`（HuggingFace 生态极常见）→ 产物中**每个 `false`** 被替换成 `[REDACTED:TOKENIZERS_PARALLELISM]`
- 任意 `*_TOKEN=1` / `*_SECRET=0` → 把产物里**所有 `1`/`0`** 替换掉，summary/归因数字直接被污染

**为什么是真问题**：`redact_secrets` 同时用于 `summary.json`、`summary.md`、`agent_stdout`、archive。`_write_gate_summary` 用 `env=source_env=os.environ` 调 redact——也就是说**用户真实 shell 环境里只要有这类变量，gate 报告就会被误伤**。这是相对 M4.0 的**功能回归风险**。

注意：放宽用于 **scan**（`_is_secret_env_name`）是安全的——扫描宁可多报。问题只在 **redact 端**：扫描要宽、脱敏要稳。

**建议**：redact 端加最小长度护栏并保留较精确的名称判定，例如
```python
if _is_secret_env_name(name) and len(value) >= 8:
    out = out.replace(value, f"[REDACTED:{name}]")
```
长度阈值可挡掉 `true/false/0/1` 这类短值；真实 key（`sk-...`、bearer token）均远超 8 字符，不受影响。

**复现**：构造 `env={"TOKENIZERS_PARALLELISM":"false", ...}`，让 fake agent `print("test passed: false alarm")`，断言 redact 后字符串里 `false` 是否被替换即可证伪。

## P2 — 真实盲点 / 应文档化或补齐

### P2-1 secret scan 对「agent 写进 workspace 的文件」既不脱敏也不扫描

`mu/eval.py:178-187` `_secret_scan_ignore_reason`：

```python
if "workspace" in parts:
    return "workspace source fixture is not a process artifact"
```

按**路径中任意一段名为 `workspace`** 就整体跳过。意图是放过「复制出来的候选/任务源码 fixture」，合理；但副作用是：**agent 在自己 cwd（即 `task_dir/workspace`）里写出的任何文件，都既不会被 `redact_secrets` 处理（runner 不持久化 workspace 文件，原样留盘），也不会被 scan 覆盖**。

即：如果模型把 `MU_API_KEY` 回显进 `calc.py`，那把 key 会**以明文留在 workspace 磁盘上**，而 `summary` 里 secret scan 仍报 **PASS**。盲点窄（正常 coding 任务不会写 key），但「PASS」高估了实际保证范围。

**建议（二选一）**：
- 文档化：在 summary/README 明确「scan 仅覆盖过程产物（stdout/stderr/validation/summary/archive），不含 agent 在 workspace 内自写的文件」。
- 或：对 workspace 文件**仅做 env-value 精确匹配**扫描（保留对 fixture 的 `pattern:sk` 豁免，避免误报假 key），既堵住真实 env key 外泄又不把 fixture 假 key 算进去。

### P2-2 新增入口 `mu/eval_gate.py` 零自动化测试

整个 `eval_gate` 模块（`run_full_gate` / `_run_dgm_smoke` / `_render_report` / `_check_detail` / `default_gate_dir` / `_fake_agent_source`）**没有任何 `tests/test_eval_gate.py`**，仅靠手工 `评测/2026-6-12-02-*` 验收。对一个新 CLI 入口而言覆盖偏薄。

其中**离线可测**的部分不少：
- `_render_report` / `_check_detail`：纯字符串渲染，可对 PASS/FAIL、skipped、error 三态做断言。
- `default_gate_dir(now=...)`：注入固定 `datetime` 验证目录名（顺带验 P3-3）。
- `_fake_agent_source()`：可 `exec`/子进程跑一遍，确认三个分支都能让对应 validator 过——这是 gate 正确性的关键，目前完全没测。
- `run_full_gate(allow_missing_model=True, ...)`：在临时 `project_root` 上跑（offline pytest 子进程较重，可用极小桩 project 或 monkeypatch 掉 `_run_offline_pytest`）+ dgm fake smoke，验证 summary/report 落盘与 `passed` 聚合逻辑。

**建议**：至少补 `_render_report` / `_check_detail` / `_fake_agent_source` / `default_gate_dir` 的离线单测；`run_full_gate` 用 `allow_missing_model=True` + monkeypatch offline pytest 做一条 happy-path。

### P2-3 `_run_dgm_smoke` 固定 `candidate_id="m4.1-smoke"`，固定 run-root 下复跑必失败

`mu/eval_gate.py:161-172` 用 `candidate_id="m4.1-smoke"`，而 `run_dgm_candidate`（`mu/dgm.py:87-89`）：

```python
candidate_root = archive / "candidates" / cid / "workspace"
if candidate_root.exists():
    raise DgmConfigError(f"candidate workspace already exists: {candidate_root}")
```

默认 gate_dir 带 `HHMMSS` 时间戳，跨次运行不冲突，所以默认路径**没问题**。但 CLI 暴露了 `--run-root`，一旦用户传**固定目录**（很自然的用法，如 `--run-root ./gate-out`），**第二次运行**时 `archive/candidates/m4.1-smoke/workspace` 已存在 → smoke 检查捕获 `DgmConfigError` 返回 `passed=False` → 整个 gate FAIL，且原因隐藏在 detail 串里，排查成本高。

**建议**：smoke 的 candidate_id 用 `"m4.1-smoke-" + 时间戳`，或在调用前清理 `smoke_dir/archive`，使 gate 对固定 `--run-root` 幂等。

## P3 — Nit / 可选

### P3-1 `run_eval_suite` 双写 summary，但缺 eval_gate 那样的解释性注释
`mu/eval.py:270-274` 先无 scan 写一次 summary，再 scan，再带 scan 写第二次（latest-summary 也连带写两次）。逻辑正确（让 summary 自身纳入 scan 视野），但 `eval_gate` 在同款双写处有注释 `# Include the generated report...`，而 `eval.py` 没有。建议补一行注释，否则后人易当成误写删掉一处。属 2× IO，无功能问题。

### P3-2 `_failure_stage` 存在不可达分支
`mu/eval.py:455-464`：

```python
if result.agent_returncode not in (0, None):
    return "agent"
if result.agent_returncode is None:   # 未 timeout 时 subprocess.run 必返回 int → 不可达
    return "agent"
```
`agent_returncode is None` 只在 `agent_timed_out=True` 时成立，而该情况已在前面 `return "agent_timeout"`。此分支为冗余死代码，删之无害。

### P3-3 `default_gate_dir` 月/日未零填充
`mu/eval_gate.py:32-35` 用 `f"{current.year}-{current.month}-{current.day}-..."` → `2026-6-12-...`，与 `eval.py` 的 `%Y%m%d` 不一致。带 `HHMMSS` 不会撞名，但 `评测/` 目录字典序排序会乱（`2026-6-2` 排在 `2026-6-12` 后）。建议统一 `%Y-%m-%d`。

### P3-4 gate 子进程仅捕获 `TimeoutExpired`
`_run` / `_run_offline_pytest` 只接 `subprocess.TimeoutExpired`。若 `pytest`/解释器缺失抛 `FileNotFoundError`，会**直接掀翻整个 gate**而非记一条 FAIL check。低概率，建议兜一层转 FAIL。

### P3-5 单跑 `python -m mu.eval` 时 `latest-summary.*` 不在 per-run scan 覆盖内
`write_eval_summary` 把 latest-summary 复制到 `run_dir.parent`，而 per-run `scan_eval_artifacts_for_secrets(run_dir)` 只扫 `run_dir` 子树，覆盖不到 parent 里的 latest-summary（仅靠 redact 兜底）。gate 场景因最终扫 `gate_dir` 整树可覆盖，故仅影响独立 `mu.eval`。属保证边界，非泄漏。

### P3-6 单文件可产出两条 hit（env + pattern）
`scan_eval_artifacts_for_secrets` 对同一文件可能既加 `env:NAME` 又加 `pattern:sk` 两条记录（测试 `test_secret_scan_...` 也固化了这一顺序）。是报告粒度问题，非 bug；但测试断言**强依赖 hit 顺序**（env 先、pattern 后），实现端若调整循环顺序会脆断，留意。

## 做得好的地方（建议保留）

- **根因修复到位**：`.resolve()` 全链路 + `--rootdir <workspace>` + 指定测试文件，两者都配了**会失败于旧实现**的针对性测试（`test_run_pytest_ignores_parent_pyproject_testpaths` 是很好的回归锚点）。
- **scan/redact 职责清晰**：scan 端宽、`ignored` 列表显式记录豁免原因，可审计。
- **fake-agent smoke 设计干净**：gate 的 DGM 冒烟用本地 fake agent，无需真 key 即可验证 archive 闭环，符合「离线可复现」基线。
- **secret-scan-then-rewrite 闭环**：让 summary/report 自身也进入扫描视野，思路正确。
