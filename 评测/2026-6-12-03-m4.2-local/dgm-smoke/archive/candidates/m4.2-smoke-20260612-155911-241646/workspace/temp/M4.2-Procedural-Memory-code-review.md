## P2 — 建议处理

### P2-1 meta-tool 超时上限 120s ＜ README 示例的内层 bash 上限 180s（示例自相矛盾）

`mu/metatool.py:22` `_META_TOOL_TIMEOUT = 120.0`，是整个 meta-tool 执行的 soft-timeout 上限（`_execute` 的 `asyncio.wait_for(..., timeout=_META_TOOL_TIMEOUT)`）。

而 **README 的 canonical 示例 spec**（`README.md` Meta-tools 节）内层用：

```text
"code": "...mu.bash(f'./.venv/bin/python -m pytest -q {target}', timeout=180)..."
```

内层 bash 申请 `timeout=180`，但外层 meta-tool 在 **120s** 就会 soft-timeout 返回，`mu._cancelled=True`。结果：**示例宣称能跑 180s 的 pytest，实际 120s 必被外层截断**，且截断后 worker 线程里的 bash 仍在跑（soft timeout 不硬停），用户看到「meta-tool timed out」却又有个孤儿子进程——体验和文档都误导。

**为何是真问题**：这是**随仓库发布的示例**，会被直接复制使用；上限关系是「内层 ≤ 外层」才合理，现在反了。

**建议（任一）**：
- 把 `_META_TOOL_TIMEOUT` 调到明显大于常见内层 bash（如 300s），或设为可配置（`MU_METATOOL_TIMEOUT`）；
- 并把 README 示例内层 timeout 收到 ≤ 上限（如 100s），保证示例自洽。

**复现**：写一个 spec `code: "mu.bash('sleep 5', timeout=180)\nmu.result('done')"`，把 `_META_TOOL_TIMEOUT` 临时调到 2s，执行 → 返回 soft-timeout 而非 done，证明外层先于内层触发。

### P2-2 meta-tool soft-timeout / 取消令牌路径无测试（安全相关代码未覆盖）

`mu/metatool.py:209-222` 的超时分支与 `_MetaToolApi.call` 的 `_cancelled` 守卫，是**安全相关**逻辑：超时后滞留 worker 线程的后续 `mu.call()` 必须被挡住，避免把工具调用注入到 agent 已经进入的**下一轮**。codeact 有等价机制；但 `test_metatool.py` 里**没有任何用例触发 `asyncio.TimeoutError` 分支**，也没验证「取消后 `mu.call` 抛 RuntimeError」。

当前 9 个测试覆盖了加载/校验/重名/冲突/readonly/reload/inner-events/prompt-hint，唯独漏了**超时与取消令牌**——恰是最易回归、后果最隐蔽的一段。

**建议**：补一条 `_META_TOOL_TIMEOUT` 调小（或注入慢 `mu.bash`）触发 soft-timeout 的测试，断言：① 返回含 "soft timeout"；② 之后 `mu.call` 抛 `RuntimeError`（可对 `_MetaToolApi` 直接单测 `_cancelled=True` → `call` 抛错，离线、无需真超时）。

### P2-3 启用 `--metatools` = 在被加载的 repo 上执行其 `.mu/metatools/*.json` 内任意 Python（信任边界，需在「快速上手」处更醒目）

`Agent.__init__` 里 `metatools=True` 即 `MetaToolManager(...).load_all()`，按**文件名 glob** 从 `.mu/metatools/` 读 spec 并把其 `code` 注册为可执行工具；默认 `allow` 下这些 code 进程内 exec、全权限、可 `import os`。

README 已写了「运行任意 Python，安全等级等同 code」——**这点做对了**。但与 code-action 有一处实质差异值得强调：**code-action 的代码是模型在 transcript 里当场写的（可见、可审）；meta-tool 的代码是预先躺在磁盘上的**。`git clone` 一个含 `.mu/metatools/evil.json` 的仓库后 `mu --metatools`，就等于运行它植入的代码——这是比 code-action 更接近「供应链」的触发方式。

注：这与 `.mu/extensions/*.py` 同性质，且 meta-tools 反而更保守（默认关、需显式 flag，而 extensions 默认 autoload）。所以**不是回归**，但鉴于触发门槛只是一个 flag，建议在 README「快速上手」段落（不只在小节里）补一句「仅对可信仓库启用 `--metatools`」。严重度：文档强化，非代码缺陷。

## P3 — Nit / 可选

### P3-1 README 示例教了「未加引号的 shell 插值」
README 示例 `code` 用 `mu.bash(f'... pytest -q {target}', ...)`，`target` 来自模型入参，未 `shlex.quote`。在 `allow` 下这不构成新权限边界（meta-tool 本就可跑任意代码），但作为**范例**会传播命令注入坏习惯（`target="; rm -rf ~"`）。建议示例改用安全拼接或加注释提示。

### P3-2 meta-tool 注册不随权限策略短路（与 extensions 不一致，但与 codeact 一致）
`load_all()` 在 `__init__` 无条件注册 spec，`readonly/workspace` 下这些 `code_exec` 工具仍出现在 `schemas()` 里被「广告」给模型，调用后才返回 permission denied——浪费 round-trip。extensions 用 `permits("load_extension")` 在 restrictive 下跳过 autoload；codeact 则和 meta-tool 一样无条件注册 `code`。即此处**与 codeact 一致、与 extensions 不一致**。注册本身只解析 JSON（无 code 执行），故**无安全问题**，仅 UX/token。可选：restrictive 策略下跳过注册（保留 `list_metatools`），三处（codeact/metatool/extension）统一更佳。

### P3-3 `MetaToolManager` / `MetaToolSpec` 未在 `mu/__init__.py` 导出
同辈 `CodeAction`、`ExtensionManager`、`make_environment` 均已导出，meta-tool 公共类未导出。小一致性问题。

### P3-4 内层 `call_id` 非唯一
`_proxied` 的 `call_id = f"metatool:{meta}:{tool}"`：同一 meta-tool 内 3 次 `mu.read` → 3 个相同 call_id。`AttributionCollector` 按 `event.name`（非 call_id）累计，故归因不受影响；但任何「按 call_id 配对 Started/Finished」的有状态渲染器在重复内层调用下可能错配。与 codeact 的 `code:{name}` 同模式，属既有约定，仅记录备查。

### P3-5 `version` 字段仅作展示
spec `version` 校验为非空串后只拼进 description（`(meta-tool v{version})`），无 semver/缓存/按版本 reload 语义。v1 可接受，记录边界即可。

## 做得好的地方（建议保留）

- **桥接忠实复用 codeact**：`_proxied` / `_MetaToolApi` / soft-timeout / 取消令牌与 code-action 同构，降低认知与维护成本。
- **校验全面且测齐**：name 正则、必填、`parameters.type=="object"`、目录内重名、与既有工具冲突、坏 spec 不致命——逐项有对应断言。
- **权限 gating 正确**：固定 `{code_exec}`，`readonly/workspace` 经 `_READONLY_BLOCK`/`_UNCONFINABLE` 整体拦截，且 `test_metatool_blocked_by_readonly_permission` 直接验证「拒绝且文件未生成」。
- **DGM 边界对称扩展**：allowlist 加 `.mu/metatools/*.json`、`_requires_exec_permission` 同步纳入、restrictive 下拒绝，均有新测试（`test_dgm_*metatool*`）。
- **离线可复现 gate**：`_run_metatool_smoke` 用 fake-model 跑通「注册→调用→内层 bash 事件→readonly 拒绝」四点闭环，且 `_render_report`/repeatable 都有单测——把 M4.1 留下的 gate 可测性补齐了。
- **采纳上轮评审**：redact 长度护栏、eval_gate 测试、fixed-root 幂等、smoke id `%f` 全部落地。
