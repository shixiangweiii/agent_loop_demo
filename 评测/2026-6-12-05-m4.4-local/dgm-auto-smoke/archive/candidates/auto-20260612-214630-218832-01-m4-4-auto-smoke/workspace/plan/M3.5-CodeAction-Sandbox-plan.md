# 实施 Plan · M3.5 — Native code-action + 可插拔安全/沙箱（v1 收口）

> 阶段来源：`docs/Python复刻Pi-Roadmap-v1.md`（v1.1）§4 M3.5。这是 **v1 最后一个阶段**。
> 经审批后**第 0 步**先落盘到 `plan/M3.5-CodeAction-Sandbox-plan.md`。虚拟环境一律 `./.venv`。
> 前序 plan 已归档 M0/M1/M2/M3。

## Context（为什么做这个）

M0–M3 完成（56 离线测试）。M3.5 落地 v1 锁定的两个 🟢 创新：① **Native code-as-action**（模型写 Python 直接当动作，一次 round-trip 组合多个工具调用 + 控制流）；② **可插拔安全/权限/沙箱层**（在 M3 子进程隔离之上加 permission 策略 + 可插拔 Environment provider）。两项都**默认关、可独立开关，关闭后退回 M3 形态**（roadmap 完成标志）。code-action 必须对 bash 有**可测量收益**（少轮数/少 token），否则即「第二套 bash」应砍。

## 关键设计决策（已锁定 / 已定）

1. **code-action = 进程内 exec**（线程 + 回调事件循环），`mu.*` 工具 API 调 `ToolRegistry.execute`（过权限策略 + 发事件）。✅ 用户锁定
   - 与 bash 的差异（即「可测量收益」来源）：**在一次 model 调用里用 Python 控制流组合多个已注册工具**（read/edit/bash/扩展工具），共享变量状态 → 把 N 轮 tool-call 压成 1 轮。
   - 代价（如实标注）：code 本身不被沙箱隔离（可 `import os` 绕过，风险等同已有的 bash）；要真隔离把 μ 跑容器里。permission 层拦的是**工具调用**（与 bash 同级的控制）。
2. **三项默认全关**（YOLO/Pi 等价）：code-action 默认关（`--code` 开）、permission 默认 `allow`、sandbox 默认 `local`。关掉即退回 M3。
3. **权限层 = 全量实现 + 全测**（纯 Python，高价值）；**沙箱 provider = 抽象 + 本地默认 + Docker 实验性可选**（docker-gated 测试，E2B/Modal 文档化可插拔点）。抽象本身是「可插拔」的交付物。
4. 保持 Pi-thin：不做受限解释器/AST 白名单、不做人审批（human-in-the-loop）Plan Mode、不做 E2B/Modal 具体实现（仅留接口）。

## 架构（复用 M1–M3，单一权限/环境钩点）

```
ToolRegistry.execute(name, args)
   ├─ policy(name, args) -> deny-reason?   # permission 钩点：gate 内置/扩展/code-action 内层调用
   └─ handler(args) 用 self._env           # env = Environment provider（local | docker）
CodeAction(registry, emitter)  → 注册 `code` 工具（仅 --code 时）
   └─ 进程内线程跑模型 Python；mu.read/write/edit/bash/call → run_coroutine_threadsafe 回事件循环
        每个内层调用 emit ToolCallStarted/Finished（可观测、计入归因）→ 走 registry → 过 policy
```

## 文件清单

**新增**：
- `mu/permission.py`：`PermissionPolicy = Callable[[str, dict], str | None]`（返回 deny 原因或 None）；`allow_all`、`read_only`（拦 write/edit/bash）、`workspace_write(root)`（拦越界写）；`make_policy(kind, **opts)`。
- `mu/codeact.py`：`CodeAction`（持 registry + emitter）；`code` 工具 schema + handler；进程内执行（`asyncio.to_thread` 跑模型代码，soft timeout）；`mu` API 对象（`read/write/edit/bash/call/log/result`，经 `run_coroutine_threadsafe` 回事件循环、emit 内层 ToolCall 事件）。
- `extensions/`（可选）补一个 code-action 范例片段到文档。

**修改**：
- `mu/environment.py`：抽出 `Environment` Protocol（`run_bash/read_file/write_file`）；`make_environment(kind, workspace=None)`（`local`→LocalEnvironment，`docker`→DockerEnvironment）；新增 **DockerEnvironment**（最小：`docker run --rm` 跑 bash、workspace 挂载，标注 experimental）。
- `mu/tools.py`：`ToolRegistry.__init__` 接 `policy=allow_all`；`execute` 在 handler 前做 `reason = self._policy(name, args); if reason: return ToolResult("Error: permission denied: ...")`。默认 allow_all → 现有行为不变。
- `mu/agent.py`：`Agent.__init__` 接 `code_action=False`、`policy=None`、`env=None`；用 `env`/`policy` 建/配 registry；`code_action` 时 `CodeAction(self.tools, self.emitter)` 注册 `code` 工具。
- `mu/cli.py` / `mu/tui.py`：flags `--code`、`--permission allow|readonly`、`--sandbox local|docker`（亦读 `MU_CODE_ACTION`/`MU_PERMISSION`/`MU_SANDBOX`），传入 Agent。
- `mu/prompts.py`：仅当 code-action 开启时附一句提示（指向 `code` 工具）。
- `mu/__init__.py`：导出 `PermissionPolicy`/`make_policy`/`CodeAction`/`make_environment`。

**新增测试**：`tests/test_permission.py`、`tests/test_codeact.py`、`tests/test_sandbox.py`（docker-gated，`shutil.which("docker")` 缺则 skip）。

## code-action 执行细节（进程内 sync↔async 桥）

- handler（async）取 `loop = asyncio.get_running_loop()`，`await asyncio.wait_for(asyncio.to_thread(self._run, code, loop), timeout)`。
- `_run`（线程内同步）：构造 `mu` 对象注入 globals，`exec(compile(code), g)`；捕获 stdout 与异常（错误转字符串）；返回 `mu.result(...)` 设的值 + 截断 stdout。
- `mu.read(path,...)` 等：`fut = asyncio.run_coroutine_threadsafe(self._proxied(name, args), loop); return fut.result()`；`_proxied` 在 loop 上 emit `ToolCallStarted`→`await registry.execute`→emit `ToolCallFinished`，返回字符串。
- 不暴露 `code` 自身（防递归）；`mu.call(name, args)` 可调任意已注册工具（含扩展）。
- soft timeout：超时返回错误（线程可能滞留，标注 limitation）。

## 复用而非新造
- `LocalEnvironment.run_bash/read_file/write_file` 即 `Environment` Protocol 的本地实现，复用其进程组清理。
- `ToolRegistry.execute` 单一钩点 gate 一切（内置/扩展/code-action 内层）。
- 内层调用复用 `events.ToolCallStarted/Finished` → 归因/渲染自动覆盖（不另造观测）。
- `asyncio.to_thread` + `run_coroutine_threadsafe`（标准库）做 sync↔async 桥，不引框架。

## 验证

**自动化（必须，无网络/无付费/无真实 LLM；docker 测试缺 docker 则 skip）：**
```
./.venv/bin/python -m pytest -q
```
覆盖：
- **权限**：`ToolRegistry(policy=read_only)` → `execute("write")`/`execute("bash")` 返回 permission denied；`execute("read")` 放行。workspace_write 拦越界写。默认 allow_all 不改变现有行为。
- **code-action 闭环 + 收益（FakeModel 离线）**：脚本化**一个** `code` 调用，其代码 `for f in files: mu.read(f)` 读 3 个文件并 `mu.result(...)` → 断言 3 个内层 read 真执行、结果正确、发了 3 对内层 ToolCall 事件、**model 仅 1 轮 code（对比 tool-call 路径需 3 轮）= 可测量收益**。
- **默认关**：未开 `--code`/code_action=False 时 registry 无 `code` 工具。
- **权限作用于 code-action**：read_only 下 code 里 `mu.write(...)` 被拒（证明 code-action 受 permission 约束）。
- 保持绿的 M0–M3 既有 56 测试。

**手工 e2e（需真实 key）：**
```
export MU_BASE_URL=...; export MU_MODEL=...; export MU_API_KEY=...
./.venv/bin/python -m mu --code "统计当前目录所有 .py 文件的总行数"     # 期望：一次 code 调用内循环 read，不是逐文件多轮
./.venv/bin/python -m mu --permission readonly "改写 README"            # 期望：write/edit 被拒，模型收到 permission denied
./.venv/bin/python -m mu --sandbox docker "echo hi 并 ls"               # 若装了 docker：bash 在容器内执行
```
